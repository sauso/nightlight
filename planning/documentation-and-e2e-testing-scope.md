# Nightlight: Documentation (with screenshots) + End-to-End Testing — Scope Document

Status: **Not started.** Planning document only, to hand to Claude Code when
work begins. Confirm current repo structure against this document before
starting.

---

## Why these two are grouped together

Manually-taken screenshots go stale the moment the UI changes, and nobody
reliably remembers to retake them. The fix is generating documentation
screenshots as a byproduct of an automated test suite that's already
exercising the real UI - so the same investment (Playwright) serves both
goals at once, and screenshots stay perpetually in sync with the actual app
rather than drifting from it.

---

## Part 1 — Documentation

**Recommended approach: a `docs/` folder in the repo, plain markdown.**
Renders natively on GitHub with zero tooling or hosting to think about.
Appropriate for the project's current size/audience (people cloning the
repo, not a large public product). A static-site generator (MkDocs Material,
Docusaurus, deployed to GitHub Pages) is a reasonable upgrade later if the
project grows a bigger audience, but not needed to start.

**Proposed structure:**
```
docs/
  README.md              - index/overview, links to the sections below
  setup.md               - Docker/Unraid installation and first-run setup
  cameras.md             - adding a camera, RTSP config (and later, ONVIF
                           discovery once that feature exists)
  android-app.md          - installing/using the companion app, the
                           three-state audio button
  screenshots/            - images referenced from the above, generated by
                           the Playwright suite (see Part 2)
```

**What generates the screenshots:** Playwright test specs that navigate to
specific app states and call `page.screenshot()` at the right moment -
e.g. "camera tile with background audio active," "add camera flow,"
"settings page." These can be a distinct, lighter-weight set of specs from
the main E2E assertions below, since their job is capturing a good-looking
state rather than verifying behavior - though it's fine for the same spec to
do both (assert correctness, then screenshot the passing state).

---

## Part 2 — End-to-end automated testing

**Recommended tool: Playwright.** Modern, strong React support, runs on
standard GitHub-hosted Linux runners with no special cost multiplier (unlike
the macOS runners needed for iOS work).

**What it can cover directly, against the real web app:**
- Adding a camera (manual entry now; ONVIF discovery flow later once built)
- The three-state audio button (On -> Off -> Background cycle) and its
  visual states
- Settings flows, temperature unit toggle, etc.
- Visual regression (screenshot diffing) as a side benefit of the same
  specs - flags unintended UI shifts from unrelated changes, on top of
  generating documentation images

**The real wrinkle: no real RTSP cameras in CI.** Solution is a synthetic
camera source - ffmpeg looping a test video file as a fake RTSP stream that
MediaMTX picks up during test runs. This gives the test suite a real (if
fake) video/audio stream to exercise the actual pipeline against - WebRTC
playback, HLS fallback, the mute/background-audio state logic - without any
hardware in the loop. Needs a small addition to the test environment's
Docker Compose setup (an ffmpeg container looping a sample video into a
path MediaMTX serves), not part of the production stack.

**What Playwright does NOT cover well: Android-native behavior.** The
foreground service actually surviving screen-off, the notification's Stop
button, background audio continuing after the app is minimized - this is
genuine hardware/OS-behavior territory. Recommended tool here is Android's
own instrumented testing (Espresso), run against an emulator.

**Note on the emulator, given prior experience:** the local Windows VM
emulator was unusable (no nested virtualization available in that
environment). GitHub Actions' own Linux runners have working KVM
acceleration, so an Android emulator running in CI is realistic even though
it wasn't viable locally - worth setting up the CI-based emulator test
before concluding Android UI/instrumented testing isn't feasible at all.

## Open questions to resolve before starting

- Where do E2E tests live - inside the existing `nightlight` repo (a
  `frontend/e2e/` or top-level `e2e/` folder) or a separate test-only
  package? Given the synthetic-camera Docker Compose addition is
  test-infrastructure rather than app code, a dedicated top-level `e2e/`
  folder with its own compose file is probably cleaner than mixing it into
  `frontend/`.
- CI trigger scope - run the full E2E suite on every push/PR, or only on
  merge to main / on a schedule? Full suite likely takes real time
  (spinning up the app + synthetic camera + browser), worth deciding this
  deliberately rather than defaulting to "every commit."
- How many/which camera states are worth screenshotting for docs purposes -
  a full inventory should probably be scoped once the docs' actual outline
  (Part 1's structure above) is finalized, so screenshot specs are written
  against real documentation needs rather than guessed at.
- Whether Android instrumented tests (Espresso) run in the same CI workflow
  as the web E2E suite, or as a separate workflow file in
  `nightlight-android` - likely the latter, given they're testing different
  repos entirely.

---

## Suggested order of work when starting

1. Stand up the synthetic RTSP camera in a test Docker Compose file - this
   unblocks everything else and is worth confirming works in isolation
   first.
2. Write a small number of Playwright specs covering the core flows (add
   camera, three-state audio button) - prove the pattern works before
   writing many more.
3. Add screenshot capture to those same specs, and use the output to write
   the actual `docs/` content (Part 1) - this ties documentation writing to
   real, current UI rather than doing it separately from testing.
4. Wire the E2E suite into GitHub Actions, deciding trigger scope
   deliberately (see open questions above).
5. Only then evaluate Android instrumented testing (Espresso) as a
   separate, later piece of work in the `nightlight-android` repo.
